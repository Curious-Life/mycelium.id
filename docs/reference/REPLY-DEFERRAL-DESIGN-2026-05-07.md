# Reply-Tool Deferral Fix â€” Design

**Date:** 2026-05-07
**Status:** Design (sweep-first protocol applied)
**Companion handoff:** [docs/SECRETS-AND-EGRESS-HANDOFF-2026-05-07.md](SECRETS-AND-EGRESS-HANDOFF-2026-05-07.md)
**Triggering incidents:** Mya missed two Telegram replies on 2026-05-07 at 08:07 UTC and 08:12 UTC. Phase 3 operator-diagnostic (`chat-fallback-skipped`) correctly fired in both cases â€” system worked, agent failed to deliver.

---

## TL;DR

The `reply` MCP tool is deferred by Claude Code's tool-listing budget ~88% of the time for Mya (14 of 16 reply calls in last 7 days required `ToolSearch select:mcp__agent-tools__reply` first). When mid-turn auto-compact strips deferred-tool schemas, the agent has to remember to ToolSearch before calling reply. Sometimes she does (07:59 â†’ success), sometimes she doesn't (08:07, 08:12 â†’ silent drops with operator-diagnostic).

The fix is two phases:

1. **Phase 1 â€” reduce tool count from 51 â†’ ~30** by retiring 19 cold-tier tools (zero calls in 7 days) + retiring 2 deprecated duplicates. Empirical evidence: moms-agent (fewer extras MCPs â†’ fewer total tools) has 18% reply-deferral rate vs Mya's 88%.

2. **Phase 2 â€” extend responseNotesSection** with explicit post-compact discipline that reinforces (a) orientation reasoning is scratchpad and structurally cannot leak (chat.js:1051-1057 already strips it), (b) tool-loading preparation is cheap and idempotent, (c) reply contains only the user-facing answer, (d) NO_REPLY remains a valid outcome.

Phase 3 (split agent-tools into core + extended MCPs) deferred until Phase 1+2 measured for 7 days.

## Revision history

- **v1 (sketch in chat).** "Pin reply non-deferred + post-compact directive." Failed pressure-test in Step 3: (a) MCP spec has no `pinned` attribute, no settings.json field, no CLI flag; (b) post-compact "first action is reply" directive risks the leak class the operator just closed.
- **v2 (this doc).** Restructured around (1) reducing total tool count to indirectly reduce deferral pressure (validated empirically by moms-agent vs Mya), (2) discipline note in responseNotesSection that preserves decision flow + explicitly closes orientation-as-reply-content path.
- **v2 â†’ v2.1 pivot during pressure-test:** Discovered chat.js:1051-1057 already strips `result.response` from HTTP response â€” orientation reasoning is structurally not deliverable via the response payload. The remaining leak concern is "agent passes orientation text as `reply({text})` argument" â€” discipline issue, not architectural. Adjusted Phase 2 framing.

## 2. Threat model

### What the design assumes about adversaries
The reply-deferral problem is **not adversarial** â€” it's an interaction between Claude Code's listing budget and Mycelium's tool count. No new attack surface introduced by the fix.

### Existing invariants preserved
- **CLAUDE.md Â§11 â€” Explicit-send only.** All user-bound delivery still goes through chokepoint routes (`/telegram/send`, `/discord/send`, `/whatsapp/send`). Phase 3's `notifyFallbackSkipped` operator-diagnostic continues to fire when agent skips explicit-send. Nothing in this design loosens those gates.
- **CLAUDE.md Â§1, Â§6 â€” No security shortcuts.** Tool retirement is removal, not bypass. Discipline note is text in a prompt; doesn't change auth, encryption, or audit.
- **Chat handler's `sanitizedResponse` at [chat.js:1051-1057](../packages/server/routes/chat.js#L1051-L1057)** â€” orientation reasoning structurally cannot reach a user via the response payload. This is the architectural backstop that makes the discipline note safe to add.

### What changes
- 21 fewer tools in the MCP listing (51 â†’ ~30) â€” reduces context budget pressure
- ~80 lines of new prompt text in responseNotesSection â€” extends existing discipline
- Reply tool description gains compact-resilience guidance â€” ~15 lines

### Risks introduced
- **Retired tool needed unexpectedly.** Mitigated: only retiring zero-call tools; tools remain in git history and can be re-added in <30 min.
- **Discipline note misread by agent.** Mitigated: text is additive to existing scratchpad framing the agent already follows; no new directive that could cause a leak.


## 7. Decision criteria for proceeding to Phase 3

**Phase 3 (split agent-tools into core + extended MCPs)** activates IF:

- After 7 days of Phase 1+2 production:
  - Mya's reply-tool deferral rate (ToolSearch select:reply / total reply calls) **stays >30%**, AND
  - At least 5 `chat-fallback-skipped` events have fired in that window

Query for assessment (run on admin):

```sql
-- Deferral rate proxy: ToolSearch select:reply / reply calls
SELECT
  (SELECT COUNT(*) FROM <jsonl-grep-equivalent> WHERE ts > 7d) AS searches,
  (SELECT COUNT(*) FROM <jsonl-grep-equivalent> WHERE ts > 7d) AS calls;

-- chat-fallback-skipped frequency
SELECT COUNT(*) FROM egress_audit
WHERE template_id = 'chat-fallback-skipped'
  AND ts > strftime('%s','now','-7 days')*1000;
```

**Success criteria (Phase 3 NOT needed):**
- Mya's deferral rate drops to <30% (matches moms-agent's 18% baseline)
- `chat-fallback-skipped` events drop to <1/day

**Failure criteria (Phase 3 activates):**
- Above thresholds breached â†’ split agent-tools into agent-tools-core (reply + 5 hottest tools) + agent-tools-extended (rest). Empirically verify whether per-MCP budgeting changes deferral behavior.

## 9. Open questions resolved during sweep

- **Is reply tool's deferral a CC bug or just budget pressure?** Pressure (51 tools exceeds budget). Verified by moms-agent's lower deferral rate at lower tool count.
- **Can we pin a tool via settings.json?** No. MCP spec v1.12.1 has no priority attribute; Claude Code has no settings field for tool pinning.
- **Does embedding schema in prompt make the tool callable without ToolSearch?** Likely no â€” Claude Code validates against its registry. Even with prompt knowledge of call shape, agent typically needs ToolSearch first. (Not 100% verified â€” would require a controlled experiment.)
- **Is orientation-reasoning leakage still possible post-Phase-3?** No, structurally â€” chat.js:1051-1057 strips `result.response` from HTTP response. The only path is `reply({text})` argument. That's a discipline issue.
- **Does prompt-section text survive auto-compact?** System prompt content is preserved by Claude Code's compaction (it summarizes conversation content, not system framing). So the discipline note in responseNotesSection persists.

## 10. Open questions deferred

- **Should `reply` tool be moved into its own MCP?** Sweep 1 confirms reply.js is fully self-contained (zero coupling). Phase 3 candidate; deferred until empirical justification.
- **Is per-MCP budgeting different from global?** Sweeps couldn't determine without source access to Claude Code's deferral algorithm. Empirical answer comes from Phase 3 if/when activated.
- **Should we promote `reply` to a non-MCP runtime primitive (e.g., `<reply>` tag parsed by runner.js)?** Bigger architectural change; defer until Phase 3 fails.
- **Should other agents (Ada, Rex, etc.) get a smaller agent-tools surface?** Their reply-tool usage is near-zero today, so no immediate pressure. Revisit when those agents handle real chat traffic.
