## 0. The shape of the experience — every touchpoint

A first-time user, from a cold machine to an explored mindscape. Each ⟵ marks a
seam this design closes; each phase tag points at the build phase that does it.

1. **Download + open the app.** A `.dmg` → drag to Applications → open. No
   terminal, no `npm`. *(The Tauri shell already exists — `src-tauri/src/main.rs`.)*
2. **First-run key ceremony.** ⟵ The app detects it has no keys, and **before any
   server boots** shows a native welcome: *"This is your vault. It lives only on
   this Mac. We'll generate two keys that encrypt everything — only you will ever
   hold them."* → **Generate** → the two keys are shown once with a copy button and
   an unmissable *"there is no recovery — back these up now"* gate → on confirm they
   are written to the macOS Keychain and the server boots. *(Phase K.)*
3. **Land in an empty, beautiful mindscape.** Signed in, no login wall *(shipped
   #24)*. First run shows a **welcome** — *"your vault is empty; import to begin"* —
   pointing at Import. ⟵ *(Phase O.)*
4. **Import.** Drag a Claude / ChatGPT export onto Import → the server detects the
   format, parses, stores encrypted, returns counts, and kicks enrichment. ⟵
   *(Phase I.)*
5. **Enrich (background).** The `:8095` service embeds + tags new messages; a quiet
   status chip shows *"enriching N…"* and settles. *(Exists; surfaced — Phase I.)*
6. **Generate the mindscape.** One button → the clustering pipeline runs with a
   **progress view**; on completion the mindscape populates. ⟵ *(Phase G.)*
7. **Explore.** Realms / territories / 3D points render from real tables; each
   territory card shows a name, an essence, and a **chronicle narrative**. ⟵
   *(Phase M for read, Phase C for the story.)*
8. **Live in it.** Library (write + publish docs), Timeline (everything that came
   in), Profile (your handle), Settings (your vault, your keys). Everything in the
   nav **works** — nothing 404s. ⟵ *(Phases N/T/P/S.)*

The seams removed across the whole journey: *no front door for keys*, *a nav full
of dead links*, *import does nothing visible*, *no way to build the mindscape from
the app*, *empty mindscape screen*, *territories with no story*.

## 1. Revision history

- **v0 — build-out doc (`UX-JOURNEY-BUILDOUT-DESIGN`).** Scoped to 5 data features
  (M/I/G/C/O). Punted the key ceremony to "stays CLI" and didn't touch the nav.
- **v1 — this doc — scope-up to the *complete* experience.** Adds: the **native
  key ceremony** (the front door), the **navigation/IA trim** (the shape), a
  **keep/cut/later decision for all 20+ routes**, and small **Timeline / Profile /
  Settings** backings. Two changes of direction:
  - **🔴 Security pivot — key ceremony is a Tauri-shell concern, NOT an HTTP
    endpoint.** A first sweep proposed a `POST /api/v1/setup/save-keys` route. That
    would put the **master key in an HTTP request body** — a direct violation of
    CLAUDE.md §4 (*master key never in HTTP headers/env/DB/logs*). Pivot: keys are
    generated by the **Rust shell** (OS CSPRNG) and written to Keychain by Rust
    **before Node boots**; the setup view is a **bundled asset**, so the key never
    crosses the network. (See §5, §6.)
  - **🟡 IA pivot — the canonical nav is the opposite of exquisite in V1.** The
    ported sidebar links to ~14 screens; in V1 **all but Library + Mindscape 404 or
    render empty** (the frontend calls cloud `/portal/*` endpoints V1 never
    implemented — confirmed by cross-referencing Sweeps 2 and 4). Pivot: trim
    primary nav to the honest V1 set and move the rest to a disabled **"Coming
    later"** group (operator's call). (See §4, §7.)

## 2. Sweep findings (consolidated, file:line)

**Sweep 1 — key ceremony & Tauri shell.**
- The Tauri shell **already exists and works**: it spawns `node src/server-rest.js`,
  waits for port 8787, opens the webview at `http://127.0.0.1:8787`, and kills the
  child on close (`src-tauri/src/main.rs:57-110`, read). It passes
  `MYCELIUM_KEY_SOURCE` (default `keychain`, `:61-68`) and **assumes the keys are
  already in Keychain** — there is **no first-run detection and no key generation**.
- Key generation + Keychain write **already exist, CLI-only**:
  `scripts/set-keys.mjs:29-37` generates two keys via
  `crypto.randomBytes(32).toString('hex')` and writes them with
  `security add-generic-password -U` (`:35-37,47-48`); the *"back up both keys
  offline… no recovery"* warning is `:74`.
- The server is **fail-closed at boot**: `resolveKeys()` → `unlock()` (`src/index.js:51-59`),
  throws on missing keys (`:56-58`), KCV mismatch throws *"vault stays locked"*
  (`src/crypto/keys.js:31-49`). First boot **creates** `kcv.json` (`keys.js:31-49`).
- Key source is pluggable: `env | keychain | 1password` selected by
  `MYCELIUM_KEY_SOURCE` (`src/crypto/key-source.js:88-104`); keys validated against
  `/^[0-9a-f]{64}$/i` (`:72-78`).
- `withGlobalTauri: true` (`src-tauri/tauri.conf.json:11`) → `window.__TAURI__.invoke`
  is available to a bundled view with no extra config. `frontendDist` is `../portal`
  (`:6-8`); no shell/keychain plugin allowlist defined.

**Sweep 2 — portal route/component inventory (the frontend's *intent*).**
- 20 app routes under `portal-app/src/routes/(app)/` + `login`. Nearly every route
  fetches a cloud `/portal/*` endpoint (e.g. wealth → `/portal/wealth/portfolios`;
  intel → `/portal/intel/report`; vitality → `/portal/vitality/snapshot`; timeline
  → `/portal/agents` + `/portal/identity`). Mindscape calls
  `/portal/onboarding/status`, `/portal/enrichment/*`, `/portal/mycelium/generate`.
- Heavy deps: three.js (Mindscape3D), leaflet + topojson + globe.gl (Intel),
  morphdom + marked + dompurify (Library) — `portal-app/package.json`.
- Orphan routes (exist, not in nav): `import`, `media`, `chat` (overlay), `contexts`.

**Sweep 3 — navigation / IA.**
- `Sidebar.svelte:48-77` (read): `coreNav` = Mycelium/Library/Timeline/Cycles/
  Agents/Profile/Connections; `spacesItem` = Spaces; `moduleNav` = Wealth/Intel/
  Body/Vitality/Activity; `fleetNav` = owner-only. Settings rendered at the bottom.
  **~14 primary targets.** It probes `/portal/connections/count` (`:22`) and
  `/portal/fleet/gate` (`:39`) — both **404 in V1**, so the badge stays 0 and Fleet
  auto-hides (the nav partly self-heals, but the module links still render).
- `Header.svelte`: sidebar toggle, mobile title, **chat toggle (Cmd+J)**, a
  pipeline-status chip, theme toggle. No search, no user menu, no command palette.
- `BottomTabBar.svelte`: mobile-only — Mycelium / Library / **Chat** / Timeline +
  "More" (opens the sidebar drawer).
- `(app)/+layout.svelte`: Header + Sidebar + main; BottomTabBar on mobile;
  ChatFloat overlay; WelcomeModal + OnboardingGuide already mounted.

**Sweep 4 — V1 server data backing (what actually exists).**
- `src/db/index.js:38-62` (read): **14 wired namespaces** — documents, messages,
  attachments, health, tasks, metrics, topology, fisher, folders, canvases, audit,
  spaces, spaceKnowledge, publicPresence. **`mindscape` and `territoryDocs` are NOT
  wired** (files exist).
- `src/portal-compat.js:36-118` (read): the **only** portal endpoints are
  `documents` (list/get/create/pin/move/delete), `folders`, `onboarding/status`.
- Tool domains exist for documents/messages/tasks/health/metrics/mindscape(search)/
  topology/fisher/ingest/internal/context. **No tool or compat surface** for
  profile, settings, timeline, activity, connections, wealth, intel, fleet, agents,
  cycles, body, vitality, media, contexts, modules.
- `db.messages` *is* wired (`:44`) → Timeline is reachable with small wiring.
  `db.spaces` *is* wired (`:54`) but no compat endpoint.

**The reconciliation (the central fact):** Sweep 2 is what the UI *tries to call*;
Sweep 4 is what the server *answers*. The canonical portal was built against the
rich cloud backend; **in V1, only Library + onboarding/status resolve**. Everything
else renders its empty state or errors. The nav must reflect V1's real surface.

## 5. The key ceremony — Tauri native first-run (Phase K)

**Goal:** a beautiful, terminal-free front door that generates the two master keys,
makes the user back them up, stores them in the Keychain, and boots the vault —
**without the key ever touching HTTP, env files, args, or logs** (CLAUDE.md §4).

### Flow

```
launch
  └─ Rust setup(): keys_present()?           ← `security find-generic-password` ×2
       ├─ YES → boot_server() → webview → portal      (today's path, unchanged)
       └─ NO  → open BUNDLED setup view (no Node yet)
                 1. Welcome: what a vault is, "only you hold the keys"
                 2. [Generate] → invoke("generate_keys") → Rust CSPRNG → {user,system}
                 3. Show both keys once · copy buttons · reveal toggle
                 4. ☐ "I've backed these up. There is no recovery."  (gate)
                 5. [Create my vault] → invoke("commit_keys",{user,system})
                       → Rust validates hex64 → `security add-generic-password -U` ×2
                       → boot_server() → navigate window to portal
```

### Rust surface (new `src-tauri/src/keys.rs`, wired in `main.rs`)

- `#[tauri::command] fn keys_present() -> bool` — both Keychain items resolve.
- `#[tauri::command] fn generate_keys() -> KeyPair` — 32 bytes ×2 from the OS
  CSPRNG (`getrandom`), hex-encoded. **In Rust memory only; not yet stored.**
- `#[tauri::command] fn commit_keys(user: String, system: String) -> Result<(),String>`
  — validate `^[0-9a-f]{64}$`; write both via
  `Command::new("security").args(["add-generic-password","-U","-a","mycelium","-s",svc,"-w",&val])`;
  on success trigger `boot_server()` + navigate the window. Mirrors
  `set-keys.mjs:35-37` exactly (same service/account names → the server's
  `key-source.js` defaults find them).
- Refactor the current `.setup()` spawn into `fn boot_server(app)` so both the
  "keys present" and "post-commit" paths call it.

### The bundled setup view

A **self-contained asset** (e.g. `src-tauri/setup/index.html` bundled via tauri
resources, or a dedicated `/setup` route in `frontendDist`) using
`window.__TAURI__.invoke` (enabled by `withGlobalTauri:true`, `tauri.conf.json:11`).
It runs **before Node** — that is what resolves the chicken-and-egg. Styled to
match the portal's glass aesthetic so the seam is invisible.

### Backup is unmissable

- Keys shown **once**, with copy buttons and a reveal toggle (default masked).
- The **"there is no recovery"** checkbox gates `commit_keys` (mirrors the CLI
  warning at `set-keys.mjs:74`).
- An **"Advanced: I use 1Password"** link instead shows the keys + the `op` refs
  from `set-keys.mjs:68-73` and writes `MYCELIUM_KEY_SOURCE=1password` for the
  spawn, rather than the Keychain. Keychain stays the golden path.

### Fallbacks (unchanged, documented)

- **CLI users:** `npm run set-keys` then `npm run start:http`/Tauri — untouched.
- **Re-install / key already present:** `keys_present()` true → straight to portal.
- **Lost keys:** by design, no recovery; the ceremony's job is to prevent this with
  the backup gate. A future "rotate keys" ceremony is deferred.

### Why this is secure (threat model delta)

- Master key generated by OS CSPRNG in the Rust process; shown once in a
  **local, bundled** webview (not served over the network); written to Keychain by
  Rust. It **never** appears in an HTTP body/header, an env file, a process arg
  (uses `args` array to `security`, not a shell string → not in `ps` via shell
  history; the value is an arg to `security` itself, same exposure as the existing
  `set-keys.mjs` — accepted, identical to today's CLI), or a log. ✓ §4.
- The Node server still boots **fail-closed** and creates/verifies the KCV
  (`keys.js:31-49`) — the ceremony only *populates* the Keychain the existing boot
  path already reads.

## 8. Edge cases — decided

- **Tauri can't reach the Keychain / `security` missing** → ceremony shows the keys
  + the manual store instructions (CLI fallback) instead of failing silently.
- **User closes the app mid-ceremony (keys generated, not committed)** → nothing was
  stored; next launch regenerates. No half-state (keys aren't written until commit).
- **`commit_keys` writes one key then fails** → report the error, re-prompt; the
  next boot's `keys_present()` requires **both**, so a partial write is treated as
  "not set up" (re-run ceremony, `-U` overwrites). Fail-closed.
- **Keys present but wrong (KCV mismatch)** → server exits (`keys.js:31-49`); the
  Tauri shell surfaces "vault failed to unlock" rather than a blank window
  (`wait_for_port` already times out at 25s, `main.rs:74`).
- **Empty vault after setup** → mindscape welcome guides to Import (Phase O).
- **Nav "Coming later" chip clicked** → no-op (disabled), optional tooltip.
- **Timeline with zero messages** → empty state, not an error.

## 10. Implementation order

**Build order (optimizes for shippable, low-risk wins + de-risking), each behind
its own `verify:*` + a Mac visual check before the next:**

1. **N — Navigation/IA trim** *(frontend-only; instantly makes the app coherent —
   nothing broken. Highest exquisite-per-risk. Do first.)*
2. **M — Mindscape read** *(lights up already-generatable topology).*
3. **I — Import** *(gets data in).*
4. **G — Generate** *(builds topology from the app).*
5. **C — Chronicles** *(makes territories read like a story).*
6. **T — Timeline**, **P — Profile**, **S — Settings** *(small backings; can batch).*
7. **K — Key ceremony** *(the front door; Rust + bundled view; Mac-gated build —
   can be developed in parallel since it's shell-side, but verified last on the Mac).*
8. **O — First-run welcome** *(the connective tissue; `showWelcome` when
   `messageCount===0`).*

> Note: **build order ≠ journey order.** The user *experiences* K → O → I → G → M →
> C; we *build* N → M → … → K → O because N de-risks everything visually, M/I/G/C
> carry the real engineering, and K is Mac-gated so it verifies last.

M/I/G/C/O detailed module shapes, LOC budgets, and endpoint signatures live in
[`UX-JOURNEY-BUILDOUT-DESIGN-2026-06-01.md` §5](UX-JOURNEY-BUILDOUT-DESIGN-2026-06-01.md);
this doc adds N, T, P, S, K.

**Proceed to the next phase when** the current phase's `verify:*` is GO **and** the
operator confirms the screen renders correctly on the Mac (`npm run portal:dev`).

## 11. Risks

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Key ceremony leaks the master key | Low | Critical | Generated/stored in Rust; bundled view (no network); never HTTP/env/log; backup gate; identical exposure to existing `set-keys.mjs` |
| Tauri first-run view can't render pre-Node | Low | High | Bundled asset via `withGlobalTauri` (`tauri.conf.json:11`); no Node dependency; Mac `cargo tauri dev` pass |
| Nav trim hides a screen the operator wanted | Low | Low | "Coming later" keeps everything visible + code stays in repo; reversible by editing one array |
| Import parser drift from real exports | Med | Med | Port + test fixtures; Claude/ChatGPT first; "format not supported yet" for others |
| Mindscape looks broken when empty | Med | Med | Graceful-empty everywhere + first-run welcome → Import |
| Backup gate ignored, keys lost | Med | High | Unmissable "no recovery" gate; copy buttons; reveal toggle; (future) re-show-once option deferred |
